## 1. 프로젝트 배경

### 1.1 왜 했는가

썸타임 어드민 대시보드(30+ 페이지)의 코드가 레거시 상태로 유지보수가 어려워지고 있었습니다. 전면 리라이트는 리스크가 크므로, **하이브리드 리라이트 전략**을 채택했습니다:

- 기존 레거시 코드는 그대로 유지
- 새로운 v2 코드를 병렬로 추가
- Feature flag(`isAdminShellV2Enabled`)로 전환 제어
- Flag OFF = 레거시 동작 (프로덕션과 동일)
- Flag ON = v2 신규 코드 동작

### 1.2 구체적으로 뭐가 좋아지는가

현재(머지 직후)는 Feature flag OFF이므로 사용자에게 달라지는 것 없습니다. 이것은 의도된 설계입니다. Flag를 페이지별로 켜면 아래 개선이 적용됩니다:

| 항목 | 이전 (레거시) | 이후 (v2 활성화 시) |
|------|-------------|-------------------|
| API 통신 | 클라이언트에서 백엔드 직접 호출 → CORS 에러 빈번 | BFF 프록시 경유 → CORS 문제 원천 차단 |
| 데이터 로딩 | 페이지마다 제각각 구현 (useEffect + useState) | React Query 통일 → 캐싱, 로딩상태, 에러처리 자동화 |
| 코드 구조 | 30개 페이지가 하나의 거대한 코드 덩어리 | 페이지별 독립 구조 → 하나 수정해도 다른 데 영향 없음 |
| 새 기능 추가 | 레거시 코드 전체 파악 필수 | v2 템플릿 따라 빠르게 추가 가능 |
| 장애 대응 | 문제 시 전체 롤백 필요 | 페이지 단위 flag OFF → 해당 페이지만 즉시 원복 |
| 타입 안전성 | API 응답 타입 미정의 | TypeScript 타입 + Zod 검증 |

**비유**: 입주자가 사는 상태에서 리모델링 준비를 완료한 것. 가설 벽을 세우고, 배관/전기를 새로 깔아두고, 스위치(flag)만 누르면 방 하나씩 새 구조로 전환 가능.

### 1.3 핵심 원칙

> **"Feature flag가 꺼져 있으면 프로덕션과 100% 동일해야 한다"**

이 원칙이 본 QA 검증의 핵심 기준입니다.

### 3.1 검증 환경

| 항목 | 프로덕션 | Preview |
|------|----------|---------|
| URL | `project-solo-gray.vercel.app` | `project-solo-git-claude-sharp-jepsen-...vercel.app` |

| Feature Flag | `false` (legacy) | `false` (legacy) |

### 3.2 검증 방식

1. **Chrome 브라우저 자동화**: 프로덕션/Preview 탭 2개를 동시에 열어 페이지별 스크린샷 비교
2. **데이터 정합성 검증**: 숫자, 목록, 상태값 등 실제 데이터 일치 여부 확인
3. **UI 요소 검증**: 버튼, 탭, 필터, 페이지네이션 등 인터랙티브 요소 존재 확인
4. **사이드바 전수검사**: JavaScript로 전체 메뉴 항목(34개)을 추출하여 1:1 비교

### 3.3 판정 기준

- **PASS**: 프로덕션과 동일하거나 의도된 개선
- **DIFF**: 차이 있으나 의도된 변경
- **FAIL**: 프로덕션 기능이 Preview에서 누락/깨짐
